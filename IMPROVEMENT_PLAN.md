# Project Memory Manager v5.0 改進計劃

## 版本目標
- **安全性優先**：消除所有安全隱患
- **真整合**：直接使用OpenClaw工具，移除模擬數據
- **架構現代化**：標準化技能結構，遵循OpenClaw最佳實踐
- **向後兼容**：現有項目無縫遷移

## 問題分析（基於Sergio的審查）

### 1. 安全性問題（高優先級）
- ❌ `subprocess`直接執行命令，沒有驗證來源
- ❌ 路徑遍歷風險：workspace檢測可能被繞過
- ❌ 缺少輸入驗證和清理

### 2. 假數據模擬（高優先級）
- ❌ `simulate_conversation_history()`：模擬數據而非真實調用
- ❌ `run_summary_update()`：mock response而非實際sessions_spawn
- ❌ 違反OpenClaw哲學：應該使用真實工具

### 3. 技術實現缺陷（中優先級）
- ❌ Token估算粗糙：`len(text) // 3`不準確
- ❌ 路徑檢測複雜脆弱：向上遍歷5層
- ❌ 缺少依賴管理：無requirements.txt
- ❌ 缺少錯誤恢復：無備份機制
- ❌ 缺少併發控制：可能文件損壞

### 4. 架構問題（中優先級）
- ❌ 技能結構非標準：Python腳本為主而非SKILL.md為主
- ❌ 配置管理分散：多處設置
- ❌ 缺少原子操作

## 改進方案

### 第一階段：安全性與真整合（Nana負責）

#### A. 移除subprocess，改用exec工具
**問題**：`version_manager.py`等多處直接調用`subprocess`執行git命令
**解決方案**：
1. 創建`git_tool.py`封裝層，提供安全的git操作接口
2. 所有git操作改為調用OpenClaw `exec`工具
3. 添加命令驗證和白名單

#### B. 實現真實工具調用
**問題**：`simulate_conversation_history()`和`run_summary_update()`使用模擬數據
**解決方案**：
1. 實現真實的`sessions_history`調用獲取對話
2. 實現真實的`sessions_spawn`調用生成摘要
3. 移除所有模擬數據生成邏輯

#### C. 路徑安全驗證
**問題**：路徑檢測可能被繞過，訪問系統其他文件
**解決方案**：
1. 添加路徑白名單驗證
2. 使用OpenClaw workspace環境變量而非向上遍歷
3. 實現路徑規範化檢查

#### D. Token估算改進
**問題**：`len(text) // 3`對中英文混合不準確
**解決方案**：
```python
def estimate_tokens(text: str) -> int:
    # 中文大概 2 chars/token，英文 4 chars/token
    import re
    chinese_chars = len(re.findall(r'[\u4e00-\u9fff]', text))
    other_chars = len(text) - chinese_chars
    return (chinese_chars // 2) + (other_chars // 4)
```

# 對話摘要系統設計

## 目標
實現「傾項目 → commit時 → AI即時記錄 → 專櫃有完整決策歷史 → git push同步」的工作流。

## 觸發條件

### 關鍵詞檢測
當用戶消息包含以下任一關鍵詞時觸發詢問：
- "commit"
- "上github" 
- "git push"
- "更新倉庫"
- "同步到github"

### 項目識別
需要從對話中識別項目名稱：
1. 用戶明確指定：「commit project-memory-manager」
2. 從上下文推斷（最近討論的項目）
3. 如無法確定，詢問：「邊個項目？」

## 流程設計

### 步驟1：觸發與詢問
```
用戶：「commit project-memory-manager上GitHub」

我：「好，準備commit project-memory-manager。要唔要同時更新專櫃摘要？」
```

**等待用戶確認**（5秒超時）：
- 如用戶話「要」或按確認：繼續步驟2
- 如用戶話「唔要」或超時：直接執行git commit + push

### 步驟2：獲取對話歷史
使用 `sessions_history` 工具：
- 讀取最近N條消息（例如最近50條）
- 根據 `last_summary_update` 時間戳過濾
- 提取與項目相關的對話

### 步驟3：生成摘要
使用 `sessions_spawn` 創建sub-agent：

```javascript
sessions_spawn({
  runtime: "subagent",
  model: "deepseek/deepseek-chat",  // 低成本模型
  task: `根據以下對話，更新項目專櫃摘要：

項目：{{項目名稱}}
最後更新時間：{{last_summary_update}}
對話範圍：{{開始時間}} 至 {{現在}}

請提取以下內容：

## decisions.md 更新
- 格式：[YYYY-MM-DD HH:MM] 決策：[內容]
- 要求：包含決策原因、考慮選項、最終選擇理由

## technical.md 更新  
- 格式：[模塊/功能]：[技術要點]
- 要求：包含實現方式、技術選擇、替代方案

## learnings.md 更新
- 格式：[洞察]：[學到咩]
- 要求：包含應用場景、經驗教訓

對話內容：
{{對話歷史}}
`,
  cleanup: "delete"
})
```

### 步驟4：更新專櫃文件
sub-agent完成後：
1. 解析生成的摘要內容
2. 分別追加到對應的 `.md` 文件
3. 更新 `project.json` 中的 `last_summary_update`
4. 可選：發送Telegram通知確認更新

### 步驟5：執行git操作
```
exec("cd {{項目路徑}} && git add . && git commit -m '...' && git push")
```

## 安全與成本控制

### 安全措施
1. **全本地運行**：sub-agent在同一部機運行
2. **無API暴露**：所有通訊內部完成
3. **可控訪問**：只讀取對話歷史，不訪問敏感文件
4. **用戶確認**：每次都需要用戶明確確認

### 成本控制
1. **低成本模型**：使用 deepseek-chat (~$0.001/次)
2. **一次性處理**：只喺commit時觸發，唔需要cron
3. **內容限長**：只處理最近相關對話（例如最近50條）
4. **可選功能**：用戶可選擇「唔要」跳過

## 錯誤處理

### 場景1：sub-agent失敗
- 重試一次（不同模型）
- 如仍然失敗，跳過摘要，只執行git操作
- 記錄錯誤到日誌

### 場景2：無法識別項目
- 詢問用戶：「邊個項目要commit？」
- 提供最近項目列表供選擇
- 如用戶取消，中止流程

### 場景3：對話歷史過長
- 智能截斷：只保留相關部分
- 使用摘要中的摘要（遞歸摘要）
- 警告用戶：「對話太長，可能遺失細節」
